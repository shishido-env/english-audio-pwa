# english-audio-pwa Roadmap

- 作成日: 2026-05-28
- ステータス: ドラフト（ユーザーレビュー待ち）
- 関連: `docs/superpowers/specs/2026-05-27-english-audio-pwa-design.md`（初期設計）

## 1. ビジョン

Anki的な英語学習PWA を、**マルチデバイス同期** を備えた形で構築する。UI/コードベース共に「最高級・保守性が高い」を目指し、機能は本当に必要なものだけに厳選する。

学習体験は次の二軸をコアとする:

- **読み上げモード**（耳学習）: 既存実装。日→英→次行 を TTS で再生
- **暗記モード**（能動学習）: 新規実装。日本語を見て思い出し、英語で答え合わせ

それ以外の高度機能（SRS、AI生成、学習統計など）は将来拡張に回す。

## 2. スコープ境界

### やる

- アカウント認証 + クラウドDBによるデッキの永続化
- マルチデバイス（PC/スマホ）からの同一ユーザーデッキ参照
- CSVアップロード経由でのデッキ取り込み（既存）
- 読み上げモード（既存）の同期対応
- 暗記モード（ミニマル）の新規実装

### やらない（明示的に除外）

- アプリからのCSVエクスポート/ダウンロード機能
- 元のCSVファイル本体の保存（パース後の `pairs` のみ保存）
- SRS（FSRS/SM-2 等）アルゴリズム
- Leitner方式の段階管理
- 学習統計ダッシュボード
- AIによるカード自動生成
- カードの手動編集UI
- デッキ共有/公開機能

## 3. サブプロジェクト構成

| # | サブプロジェクト | 順序 | 概要 |
|---|---|---|---|
| 1 | 認証 + クラウド同期基盤 | 先行 | ログイン / クラウドDB / 既存デッキを DB ベースに切替 / マルチデバイス対応 |
| 2 | データモデル確定 | 1と一体 | users / decks / cards / review_history のスキーマ。マイグレーション運用 |
| 3 | 暗記モード（ミニマル） | 1-2 完了後 | 日本語表示→思い出す→英語表示→「覚えた / まだ」記録、未習優先+ランダム出題 |

| 5 | 将来拡張 | 後日（別ロードマップ） | SRS、AI生成、学習統計など |

サブプロジェクト 1-2 は密結合のため同一の設計ドキュメント・PR群で進める。サブプロジェクト 3 以降は独立した spec/plan サイクルで進める。

## 4. 機能概要（コア3機能）

### 4.1 CSVアップロード

- 外部（ChatGPT等）で作成した日本語/英語2列CSVをアップロード
- パース結果を `pairs: { ja, en }[]` としてクラウドDBに保存
- 元のCSVファイル本体は保存しない（再アップロードで対応）

### 4.2 読み上げモード（既存機能の同期対応）

- 既存の `usePlayer`, `useIntervals`, `usePracticeMode` を維持
- データソースを localStorage から **クラウドDB（localStorageキャッシュ付き）** へ移行
- オフライン時はキャッシュで再生継続

### 4.3 暗記モード（ミニマル / 新規）

- 出題: 日本語を表示
- ユーザーが思い出す
- 「答えを見る」操作で英語表示 + 音声再生（任意）
- 「覚えた / まだ」ボタンで正誤を記録
- 出題順: 未習優先 + ランダム（SRSアルゴリズムは入れない）
- 保存対象: カードごとの正誤履歴のみ

## 5. データモデル方針

詳細スキーマはサブプロジェクト 2 で確定するが、現時点の想定:

```
users
  id, email, created_at, ...

decks
  id, user_id, name, imported_at, created_at, updated_at

cards
  id, deck_id, ja, en, position

review_history
  id, user_id, card_id, result ('remembered' | 'forgot'), reviewed_at
```

- `pairs` は `cards` テーブルに正規化して格納（順序保持のため `position` 列）
- 暗記モードの「最後にどう答えたか」は `review_history` の最新行で表現
- 将来の SRS 拡張時は `cards` または別テーブルに `interval` `ease` 列を追加する余地を残す

## 6. 開発運用ルール（エンジニアリングプラクティス）

### ブランチ戦略

- master を保護されたメインライン
- フィーチャーごとに `feat/<topic>`, `docs/<topic>`, `fix/<topic>` ブランチを切る
- PR レビュー後に master へマージ（squash推奨、コミット履歴を整える）

### ドキュメント

- 各サブプロジェクトに対し以下を作成しgit管理:
  - `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md` 設計（spec）
  - `docs/superpowers/plans/YYYY-MM-DD-<topic>-implementation.md` 実装計画
- ロードマップ更新時はこのファイルを直接編集してコミット

### コード品質

- 型ヒント必須、`strict: true` を維持
- `lib/`, `hooks/` は TDD（テストファースト）
- `components/` は手動確認（snapshot/visual test は不要）
- ESLint / Prettier をパイプラインで通す（既存設定を維持）

### コミット粒度

- 1 PR = 1 サブタスク粒度。レビュー可能な大きさに保つ
- コミットメッセージは Conventional Commits を継続（`feat:`, `fix:`, `docs:`, `chore:` 等）

### 設計→計画→実装サイクル

各サブプロジェクトで以下を順に実施:

1. brainstorming（このスキル） → spec ドキュメント
2. writing-plans → plan ドキュメント
3. executing-plans → 実装
4. requesting-code-review → レビューを依頼
5. finishing-a-development-branch → マージ判断

## 7. 既存資産と活用方針

| 既存資産 | 扱い |

| localStorage 各キー | キャッシュ層として残す。サーバが正データ |
| 既存テスト（`*.test.ts`） | 維持 |

## 8. サブプロジェクト1で確定すべき未決事項

詳細設計フェーズで以下を決定する:

- フロント基盤: **Vite継続 + Vercel Functions API** vs **Next.js App Router 移行**
- 認証プロバイダ: Clerk / Auth.js / Supabase Auth など
- DB: Neon Postgres / Supabase など（Vercel Marketplace 経由を優先）
- ORM: Drizzle / Prisma
- API スタイル: REST / tRPC / Server Actions（Next.js移行時）
- 同期戦略の詳細: 双方向同期は当面不要、サーバ一方向で十分か
- オフライン書き込みの扱い: オフライン中の暗記モード結果をどうキューイングするか

確定事項（このロードマップ時点）:

- 元CSVファイル本体は保存しない
- マルチデバイス同期はマスト
- localStorage はキャッシュ用途で継続

## 10. 想定リスクと方針

- **大きな技術スタック変更（Next.js移行）の負債**: サブ1の設計段階で「移行する/しない」を一発判断。判断後は揺らさない
- **既存機能の劣化**: サブ1完了時点で読み上げモードの動作確認を必須にする
- **オフラインPWAの劣化**: PWAキャッシュ戦略をサブ1の spec に明記する
- **複数端末でのデータ競合**: 当面は最終書き込み優先（last-write-wins）で許容。本格的な競合解決は将来検討
